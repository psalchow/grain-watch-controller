# SQLite Database Integration — Design

**Date:** 2026-06-02
**Status:** Approved (pending review)
**Author:** brainstorming session

## Goal

Introduce a lightweight relational database to the Grainwatch BFF for persisting users and stock configuration. Time-series sensor data remains in InfluxDB. The new database also reserves room for future ventilation status monitoring (out of scope for this iteration).

## Decisions Summary

| Topic | Decision |
|-------|----------|
| Database | SQLite via `better-sqlite3` |
| ORM / Query Layer | Drizzle ORM + `drizzle-kit` |
| DB file location | `backend/data/grainwatch.db` (existing Docker volume) |
| Migration of existing `users.json` | None — fresh start with default admin |
| Stock management API | Out of scope; seed only |
| Test strategy | In-memory SQLite (`:memory:`) per test |

## Architecture

### Layering

```
Controllers
    ↓
Services (UserService, StockService)
    ↓
Repositories (UserRepository, StockRepository)
    ↓
Drizzle schema + connection (db.ts)
    ↓
better-sqlite3 (WAL mode, foreign keys ON)
```

The repository layer abstracts SQL access. Services keep business rules (hashing, validation, authorisation). A later switch to PostgreSQL only requires changing the Drizzle dialect and reviewing the repositories.

### Directory Layout

```
backend/src/
├── db/
│   ├── index.ts                  # Connection singleton
│   ├── schema.ts                 # Drizzle table definitions
│   ├── migrate.ts                # Migration runner (called from bootstrap)
│   ├── migrations/               # SQL files generated by drizzle-kit
│   └── repositories/
│       ├── user.repository.ts
│       └── stock.repository.ts
```

### Environment Variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `DATABASE_PATH` | `data/grainwatch.db` | Path to SQLite file (relative to cwd or absolute) |
| `DATABASE_URL` | _(unset)_ | If set, overrides path. `:memory:` used in tests. |

`USERS_FILE_PATH` is removed from `config/index.ts`, `docker-compose.yml`, `.env.example`.

## Schema

### `users`

| Column | Type | Constraints |
|--------|------|-------------|
| `id` | TEXT | PRIMARY KEY |
| `username` | TEXT | UNIQUE NOT NULL |
| `password_hash` | TEXT | NOT NULL |
| `email` | TEXT | NULLABLE |
| `role` | TEXT | NOT NULL, CHECK in (`admin`, `viewer`) |
| `created_at` | TEXT | NOT NULL (ISO 8601 string) |
| `active` | INTEGER | NOT NULL DEFAULT 1 (SQLite boolean) |

### `user_stock_access`

Many-to-many relation between users and accessible stocks. Wildcard access is represented by a single row with `stock_id = '*'`.

| Column | Type | Constraints |
|--------|------|-------------|
| `user_id` | TEXT | FOREIGN KEY → `users.id` ON DELETE CASCADE |
| `stock_id` | TEXT | NOT NULL |
| PRIMARY KEY | (`user_id`, `stock_id`) | |

Index: `user_stock_access(stock_id)` for reverse lookups.

### `stocks`

| Column | Type | Constraints |
|--------|------|-------------|
| `id` | TEXT | PRIMARY KEY |
| `name` | TEXT | NOT NULL |
| `description` | TEXT | NULLABLE |
| `device_count` | INTEGER | NOT NULL |
| `device_group` | TEXT | NOT NULL |
| `device_prefix` | TEXT | NOT NULL |
| `has_humidity` | INTEGER | NOT NULL (boolean) |
| `active` | INTEGER | NOT NULL DEFAULT 1 |
| `created_at` | TEXT | NOT NULL |

## Boot Sequence

In `bootstrap.ts`, after config load and before HTTP server start:

1. Open SQLite connection. Apply `PRAGMA journal_mode = WAL` and `PRAGMA foreign_keys = ON`.
2. Run migrations using `drizzle-orm/better-sqlite3/migrator`. Migration state is tracked in the `__drizzle_migrations` table managed by Drizzle.
3. Seed stocks (idempotent: `INSERT ... ON CONFLICT(id) DO NOTHING`) with the values currently hard-coded in `stocks.controller.ts`:

   - `grain-watch-1` / Halle 8 / corn-watch-1 / prefix `1` / hasHumidity true / active true
   - `grain-watch-2` / Halle 7 / corn-watch-2 / prefix `2` / hasHumidity false / active false

4. Call `UserService.initializeDefaultUsers()`. If the `users` table is empty, create `admin / changeme123` with wildcard stock access.

Migration failure or DB-file unwritable → boot fails with a clear error; HTTP server does not start.

## Migrations Workflow

- Developer edits `db/schema.ts`.
- `npm run db:generate` (alias for `drizzle-kit generate`) creates a new SQL file in `db/migrations/`.
- Both schema and migration file are committed together.
- On next boot the migration is applied automatically.

## Service & Controller Changes

### `UserService`

- Drop file I/O (`loadUsers`, `saveUsers`, `usersCache`, `getAbsolutePath`, `ensureDirectoryExists`).
- Drop `filePath` constructor parameter.
- Accept `UserRepository` via constructor (DI for tests).
- Public API unchanged: `findUserByUsername`, `findUserById`, `createUser`, `updateUser`, `deleteUser`, `canAccessStock`, `initializeDefaultUsers`, `getAllUsers`, `toUserProfile`.
- `stockAccess: string[]` mapping: repository joins `user_stock_access` and returns a string array. Updates to `stockAccess` happen inside a transaction (DELETE old + INSERT new).
- Rename `UserServiceError` code `FILE_ERROR` → `DB_ERROR`.

### `StockService` (new)

- Methods: `listStocks(): Promise<Stock[]>`, `getStock(id: string): Promise<Stock | null>`.
- Replaces the in-controller `STOCK_METADATA` constant.

### `stocks.controller.ts`

- Inject `StockService`; remove `STOCK_METADATA`.
- Lookup paths (`Object.keys(STOCK_METADATA)`, `STOCK_METADATA[stockId]`) are replaced with service calls.

## Repository Interfaces

```typescript
interface UserRepository {
  findById(id: string): Promise<User | null>;
  findByUsername(username: string): Promise<User | null>;
  findAll(): Promise<User[]>;
  insert(user: User): Promise<void>;
  update(id: string, patch: Partial<User>): Promise<void>;
  delete(id: string): Promise<boolean>;
}

interface StockRepository {
  findById(id: string): Promise<Stock | null>;
  findAll(): Promise<Stock[]>;
  upsertMany(stocks: Stock[]): Promise<void>; // seed-friendly
}
```

`User` includes `stockAccess: string[]` re-hydrated from the `user_stock_access` table.

## Testing

- New helper `tests/setup/db.ts` exports `createTestDb()` returning a fresh in-memory Drizzle instance with migrations applied.
- Existing `tests/services/auth/user.service.test.ts` is ported: replace temp-JSON-file setup with `createTestDb()` and `new UserService(new UserRepository(db))`.
- New `tests/services/stock.service.test.ts`: cover `findAll`, `findById`, and seed idempotency.
- Integration tests in `tests/controllers/` keep their structure; wiring switches to repositories.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `better-sqlite3` is a native module (node-gyp) | Ensure multi-stage Dockerfile has `python3 make g++` in builder, or rely on prebuilt binaries shipped by the package. Verify Docker build succeeds before merging. |
| WAL mode creates `.db-wal` / `.db-shm` sidecars | Document that backups must include sidecars, or checkpoint before snapshot. Note added to `backend/CLAUDE.md`. |
| Synchronous API blocks event loop | Acceptable at current scale (small relational dataset). Revisit only if a long-running query appears. |
| Container file permissions | Verify the runtime user can write to `/app/data`. Document in deployment notes. |
| Migration race in multi-replica deploy | Out of scope (single-instance deploy). Postgres migration is the answer if/when scaling out. |

## Out of Scope

- Stock CRUD HTTP endpoints (POST / PATCH / DELETE).
- Admin UI for stock management.
- `ventilation_status` table — added in a follow-up when the feature is built.
- Importing data from existing `users.json` (fresh start decided).
- InfluxDB integration changes.
- PostgreSQL migration.

## Acceptance Criteria

1. `npm run dev:backend` starts, the DB file is created, migrations run, and the default admin is provisioned.
2. `POST /api/v1/auth/login` with `admin` / `changeme123` succeeds against the new DB.
3. `GET /api/v1/stocks` returns the two seeded stocks (`grain-watch-1`, `grain-watch-2`) with metadata identical to the previous hard-coded values.
4. `backend/data/users.json` and `backend/data/test-users.json` are removed from the repository.
5. `STOCK_METADATA` is removed from `backend/src/controllers/stocks.controller.ts`.
6. All existing tests pass; new tests cover repositories and seed idempotency.
7. `docker compose up` starts successfully; the DB persists at `backend/data/grainwatch.db` between restarts.
8. `npm run typecheck` and `npm run lint` are clean.
